1 Template Builder Device — Patch Building

Build
Initial patch structure

Built the Template Builder M4L device with the following objects wired together:

  • [js stemforge_template_builder.js] — the brain, creates tracks via LOM
  • [button] — triggers the build via bang
  • [textedit] — status display, shows build progress
  • [print] — logs to Max Console for debugging

Wiring: [button] outlet 0 → [js] inlet 0. [js] outlet 0 → [textedit]. [js] outlet 1 → [print].

Bug
textedit "doesn't understand" error

Status messages sent to the [textedit] object via outlet(0, msg) produced a "textedit: doesn't understand" error. The textedit object does not accept raw strings — it needs a set prefix.

Fix
outlet(0, "set", msg) prefix

Changed the status() function from outlet(0, msg) to outlet(0, "set", msg). The "set" message tells textedit to replace its contents with the following text. This pattern is used in both JS files.

function status(msg) { outlet(0, "set", msg); // was: outlet(0, msg) post(msg + "\n"); }

2 The Browser API Dead End

Attempt
live_app browser — programmatic device loading

The original plan was to load devices (VST3 plugins, Ableton effects) programmatically onto the template tracks. First attempted the live_app browser approach:

var browser = new LiveAPI("live_app browser"); // Result: "attribute doesn't exist on Application"
Failure
app.get("browser") — also failed

Tried an alternative approach via the Application object:

var app = new LiveAPI("live_app"); app.get("browser"); // Result: "'Application' object has no attribute 'browser'"
Discovery
Browser was NEVER in the LOM

After extensive research through Ableton's Live Object Model documentation and community forums, the conclusion was clear: the Browser object has never been exposed in the LOM. The original script's approach was based on incorrect assumptions about the API. There is no browser property on the Application object, and there never has been.

Research
Track.insert_device(name) — Live 12.3+ only

Track.insert_device(name) was added in Live 12.3, but it only works for native Ableton devices (Compressor, EQ Eight, Reverb, etc.), not for VST3 plugins (Decapitator, LO-FI-AF, EchoBoy, etc.). Additionally, the user's build of Live was version 2025-11-03, which predates the Live 12.3 release of November 25, 2025.

Pivot
Script creates tracks, user drags devices manually

Given that programmatic VST3 loading is impossible via LOM, the Template Builder was pivoted to a simpler role:

  • Creates all 7 tracks with correct names and colors via LOM
  • Prints a device checklist to the textedit — lists exactly which devices to drag onto each track
  • User drags devices from the Browser manually

The Template Builder is a one-time setup tool. Since you only run it once, manual device loading is acceptable. The real automation payoff is in the Loader device, which runs every time you process a track.

3 Loader Device — Patch Building

Build
More complex patch with more objects

The Loader device required a more involved patch than the Template Builder. Objects:

  • [loadbang] — fires on device load to set up paths and pipeline config
  • Three message boxes for setProcessedDir, setPipelinesDir, loadPipeline default
  • [button] — "Load Latest" manual trigger
  • [toggle] + [select 1 0] — watch mode on/off with routing
  • [startWatch] and [stopWatch] message boxes
  • [textedit] — status display
  • [print SFLoader] — completion log
Message boxes need [M] key, not [N]. In the Max editor, N creates an object box (plain rectangle). M creates a message box (has a flag/tab on the right side). The setProcessedDir, setPipelinesDir, loadPipeline, startWatch, and stopWatch messages must all be message boxes. Using object boxes instead will produce cryptic errors.
Gotcha
[select 1 0] has 3 inlets and 3 outlets

[select 1 0] is deceptive — it creates 3 inlets and 3 outlets, not 2. You must connect the [toggle] to inlet 0 (the leftmost). Outlet 0 (left) fires when input is 1, outlet 1 (middle) fires when input is 0, outlet 2 (right) fires for anything else. We only use outlets 0 and 1.

Build
Presentation mode for Ableton device chain

Max patches have two views: Patching view (where you wire things) and Presentation view (what the user sees in Ableton's device chain). To make controls visible in Ableton:

  • Right-click each UI object (button, toggle, textedit) → Add to Presentation
  • Switch to Presentation view — arrange the controls in a usable layout
  • Only objects added to Presentation appear in Ableton's device rack
  • Non-UI objects (loadbang, select, message boxes) stay in Patching view only

4 Loader Device — JS Bugs Fixed

Bug
Java interop doesn't work in Max 8/9's V8 engine

The original home directory detection code used Java interop:

var home = Packages.java.lang.System.getProperty("user.home"); // Max's V8 JavaScript engine does NOT have Java interop // This was a holdover from Max 5/6 era which used Rhino (Java-based JS)

Replaced with a message-based approach: the setProcessedDir and setPipelinesDir messages send the full paths from the Max patch, avoiding any need to detect the home directory in JS.

Bug
File.readdir() doesn't exist in Max JS

Attempted to list directory contents using File.readdir() to find the latest stems.json. This method does not exist in Max's JavaScript environment. Replaced with the Folder API, which also had its own issues (see the File Path Saga below).

Bug
outlet(0, msg) — same textedit fix

Same issue as the Template Builder. Changed outlet(0, msg) to outlet(0, "set", msg) in the Loader's status() function.

Bug
ClipSlot.create_clip(filePath) — wrong API

Originally used ClipSlot.create_clip(filePath) to load audio. This is wrong — create_clip() takes a beat length (a number), not a file path. It creates an empty MIDI clip of the specified length.

csAPI.call("create_clip", stemInfo.wav_path); // create_clip takes a beat LENGTH, not a file path // e.g., create_clip(4.0) creates a 4-beat empty clip

At this point, we believed LOM could not load audio at all. This turned out to be wrong — see the Audio Loading Breakthrough below.

Bug
device.call("load_device", ...) — doesn't exist

Attempted to load audio files via a device API call. load_device does not exist in the LOM. This was another dead end in the search for programmatic audio loading.

5 The File Path Saga

Root Cause
Max's File object CANNOT open bare POSIX paths

This was the single most time-consuming bug. Max's File object on macOS does not understand standard POSIX paths like /Users/zak/stemforge/.... It requires the "Macintosh HD:" volume prefix:

var f = new File("/Users/zak/stemforge/processed/hey_mami/stems.json"); // f.isopen === false — cannot open var f = new File("Macintosh HD:/Users/zak/stemforge/processed/hey_mami/stems.json"); // f.isopen === true — works

This was the root cause of ALL "Cannot open" and "No stems.json found" errors during development.

Fix
toMaxPath() helper function

Created a helper that prepends the volume prefix to any POSIX path:

function toMaxPath(posixPath) { var p = String(posixPath); if (p.indexOf("/") === 0) { return "Macintosh HD:" + p; } return p; }

This function is called before every new File() and new Folder() operation.

Bug
File constructor's "TEXT" typelist was filtering out JSON

The original File constructor used a typelist parameter to filter file types:

var f = new File(maxPath, "read", "TEXT"); // "TEXT" typelist filtered OUT .json files — they don't match "TEXT" type var f = new File(maxPath, "read"); // No typelist — reads any file
Bug
String() coercion needed on all path arguments

When Max sends a message like setProcessedDir /Users/zak/stemforge/processed to the [js] object, the arguments arrive as Max atoms, not JavaScript strings. Operations like string concatenation can silently fail or produce garbage. Fixed by wrapping all incoming arguments in String():

function setProcessedDir(dir) { PROCESSED_DIR = String(dir); // was: dir }
Fix
readstring(65536) instead of readline()

readline() reads one line at a time and is unreliable for JSON files that may contain long lines. Switched to readstring(65536) which reads up to 64KB at a time in a loop until EOF:

function readFileContents(path) { var maxPath = toMaxPath(path); var f = new File(maxPath, "read"); if (!f.isopen) return null; var raw = ""; while (f.position < f.eof) { raw += f.readstring(65536); } f.close(); return raw; }
Bug
Folder object also needs "Macintosh HD:" prefix

The Folder API (used for directory listing in findLatestManifest) has the same path format requirement as File. Added toMaxPath() calls to all new Folder() invocations.

6 Freeze / Unfreeze Chaos

Discovery
Freezing bundles JS into the .amxd

When you Freeze a Max for Live device (File → Freeze Device), it bundles the JavaScript file inside the .amxd binary. After that, edits to m4l/stemforge_loader.js have NO effect on the frozen device. The device continues using the bundled copy.

Discovery
Unfreezing copies JS to a different location

When you Unfreeze a device, Max copies the bundled JS to a project folder:

/Users/zak/Documents/Max 9/Max for Live Devices/StemForgeLoader Project/code/stemforge_loader.js

The device now reads from that code/ subfolder, NOT from the original m4l/ folder. To apply JS changes, you had to manually copy the updated file to the project's code/ folder after each edit.

Problem
Multiple discarded and duplicate copies

Repeated freeze/unfreeze cycles created multiple project folders and "discarded" copies. At one point there were three copies of the JS file in different locations, none of which were the current version. Extremely confusing.

Lesson learned: During active development, keep the device unfrozen. Save the .amxd in the same folder as the .js file. Only freeze once development is complete. If you need to edit the JS, unfreeze, copy your updated file to the project's code/ subfolder, then re-freeze when done.

7 The Audio Loading Breakthrough

Belief
Originally believed LOM could NOT load audio

After the create_clip() and load_device() failures, the working assumption was that the LOM simply could not load audio files into clip slots. The Loader was designed to print drag-and-drop instructions to the textedit, and the user would manually drag stems from the Browser.

Breakthrough
ClipSlot.create_audio_clip(file_path) EXISTS in Live 12+

After the user pushed back with additional research, we discovered that ClipSlot.create_audio_clip(file_path) was added in Live 12. This method takes a POSIX file path and creates an audio clip from the file:

var csAPI = new LiveAPI("live_set tracks " + trackIndex + " clip_slots " + slotIndex); csAPI.call("create_audio_clip", "/Users/zak/stemforge/processed/hey_mami/drums.wav"); // Takes a POSIX path — NOT "Macintosh HD:" format! // Opposite convention from the File object!

Critical detail: create_audio_clip() takes a standard POSIX path (/Users/zak/...), NOT the "Macintosh HD:" format that the File object requires. This is the opposite convention.

This was the key breakthrough. Full automation of stem loading became possible — the Loader could now load audio, set BPM, name clips, enable warping, and set loop mode, all without any manual drag-and-drop.

8 Track Matching Evolution

v1
Duplicate template tracks, rename them

The first approach duplicated template tracks for each song, creating new tracks named like hey_mami | drums, hey_mami | bass, etc. Problem: this created new tracks every time, cluttering the session. Over multiple loads, the track count would balloon.

v2
Load stems onto EXISTING tracks by name matching

The user wanted stems loaded onto the existing template tracks, not duplicated ones. The Loader now finds tracks by matching the stem name against the track name.

Implementation
findTrackBySuffix() with multi-pass matching

The findTrackBySuffix() function uses a multi-pass approach to match stem names to tracks:

  • Pass 1 — exact suffix: stem "drums" matches track name ending with "drums" (case-insensitive)
  • Pass 1 — pipe pattern: checks for "| drums" anywhere in the track name
  • Pass 1 — last word: splits on spaces/pipes, checks if last word matches
  • Pass 2 — loose match via STEM_TRACK_MAP: stem "drums" matches tracks containing "drums" or "drum"; stem "other" matches "texture"; stem "synthesizer" matches "synth" or "texture"
Implementation
findEmptyClipSlot() — first empty slot

Once the target track is found, findEmptyClipSlot() iterates through the track's clip slots looking for the first one without a clip. This allows loading multiple songs onto the same template tracks — each goes into the next empty slot.

9 Text Input for Track Names

Build
textedit input chain for named loading

Added a text input field so the user can type a track name (e.g., hey_mami) instead of only using "Load Latest". The wiring chain:

[textedit] → [route text] → [prepend load] → [js stemforge_loader.js]
  • [textedit] outputs messages prefixed with text (e.g., text hey_mami)
  • [route text] strips the text prefix, passing through just the name
  • [prepend load] prepends load so the JS receives load hey_mami
Fix
Clearing input after submission

Used [t l b] (trigger list bang) between route and prepend, plus a [set] message to clear the textedit after the name is submitted. Without this, old text would accumulate and get sent with the next submission.

Fix
load() takes first argument only

The load() function in the JS takes only the first argument from the message to avoid accumulation issues:

function load() { var args = arrayfromargs(messagename, arguments); var name = args.length > 1 ? String(args[1]) : ""; // Only uses args[1] — ignores any extra words }
Detail
CLI converts folder names to snake_case

The CLI's to_snake_case() function strips leading track numbers (e.g., "01 Hey Mami" becomes "hey_mami"), lowercases everything, and replaces non-alphanumeric characters with underscores. The user must type the snake_case name in the Loader's text input.

10 Clip Properties

Implementation
Clip naming, warping, and looping

After create_audio_clip() creates the clip, the Loader configures it:

  • Name: song_name-stem_part (e.g., hey_mami-drums)
  • Warping: auto-enabled via clipAPI.set("warping", 1)
  • Warp mode: set from the pipeline config (e.g., beats for drums, tones for vocals)
  • Looping: enabled via clipAPI.set("looping", 1)
var clipName = manifest.track_name + "-" + stemInfo.name; clipAPI.set("name", clipName); clipAPI.set("warping", 1); clipAPI.set("warp_mode", WARP_MODES[stemConfig.warp_mode]); clipAPI.set("looping", 1);
Change
Removed "already loaded" check

An earlier version skipped loading if a clip with the same name was already on the track. This was removed because the user wants to reload freely — overwriting the previous stem or loading into the next empty slot. The lastLoadedManifest check is reset before each explicit load request.

11 Python Environment Issues

Not M4L-specific, but these were encountered during the same session while getting the CLI working.

Problem
conda base had Python 3.8 — too old

The conda base environment was running Python 3.8, which is too old for the StemForge CLI (requires 3.11+ for type hint syntax like list[str]).

Problem
pyenv 3.11 without shell integration

pyenv had Python 3.11 installed but the shell integration was not set up. Needed to add eval "$(pyenv init -)" to the shell profile so that python resolved to the pyenv-managed version.

Problem
llvmlite build failures

llvmlite (a dependency of numba, which is a dependency of librosa) failed to build from source. Fixed by using the --no-build flag to install pre-built wheels:

uv pip install llvmlite numba librosa --no-build
Problem
x86_64 Python on ARM Mac (Rosetta)

The user was running an x86_64 (Intel) Python via Rosetta on an M2 Mac. This caused binary wheel mismatches and build failures for native libraries. Solution: switch to an ARM-native terminal and reinstall Python via pyenv for arm64.

Detail
uv for package management

Used uv instead of pip or conda for faster, more reliable package management. uv venv to create the virtualenv, uv pip install -e ".[local]" for editable install.

Problem
ffmpeg had stale library links

Homebrew's ffmpeg had stale dylib links after a system update. Fixed with:

brew reinstall ffmpeg
Problem
Demucs couldn't read .m4a files

Demucs uses torchaudio which could not read .m4a files directly. Required converting to WAV first:

ffmpeg -i track.m4a track.wav
Problem
torchcodec missing on M2 Mac

torchcodec (an optional dependency for torchaudio) was not available for Apple Silicon. Required a separate pip install or skipping the codec entirely and relying on ffmpeg for audio I/O.

TL;DR Key Lessons

  • LOM has no Browser API. You cannot programmatically load VST3 plugins onto tracks. Accept it and move on.
  • Max's File object needs "Macintosh HD:" prefix. POSIX paths don't work. Create a toMaxPath() helper.
  • LOM's create_audio_clip() uses POSIX paths. Opposite convention from File. Don't prepend the volume name.
  • String() coerce all Max message arguments. They arrive as atoms, not JS strings.
  • readstring(65536) not readline(). More reliable for JSON files.
  • outlet(0, "set", msg) for textedit. Raw strings cause "doesn't understand" errors.
  • Don't freeze during development. Keep the device unfrozen. Copy JS to the project's code/ folder.
  • Message boxes need M key, not N. N creates object boxes.
  • [select 1 0] has 3 inlets/outlets. Use inlet 0 (leftmost).
  • Convert .m4a to .wav before processing. Demucs/torchaudio can't read .m4a.
  • Use --no-build for llvmlite. Building from source fails on most systems.
