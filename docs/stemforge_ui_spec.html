COMPREHENSIVE SPEC · v3-device + workflow

StemForge — UI Design & Workflow

A single Max for Live device that drops onto an audio track, reads Ableton's warp-analysis through the LOM, splits stems via ONNX/CoreML, slices at bar boundaries, lands audio into pre-templated tracks on the shared session clock, and exposes curated bars on a MIDI-mappable pad grid.

00The two-dimensional brief

You asked for two things: a UI design for the device, and a coherent spec for how the workflow lays out tracks and clips on the shared session clock. This document treats them as one problem — the UI is the workflow, made visible.

Dimension 1 · Device UI

The .amxd patch a user sees in a device slot. v0 spec says 400×260 with a drop target + two dropdowns + a split button. v3 ambition (per specs/m4l-integrated-forge-device.md) pushes to a waveform, spectroscope, pad grid, stem mixer. This spec designs the v3 destination and flags which subset is the v0 floor.

Dimension 2 · Session choreography

What happens inside the Live set around the device when FORGE fires: template-track duplication, clip placement, warp-mode per stem, BPM propagation, beat-chop landing in Simpler, and the tempo contract that lets curated bars from different source tracks play in time together.

Typography & rhythm

  • Face: JetBrains Mono (primary) with SF Mono / Fira Code fallback. Max itself defaults to Ableton Sans; we override per-element fontname to Menlo (the closest shipped monospace Max guarantees). No prose body text — this is a control surface.
  • Sizes: 8px ALL-CAPS tracking-0.15em labels; 9.5–10.5px values; 14px for the FORGE wordmark; 11px for the title bar. Nothing else.
  • Letter-spacing: generous on labels (0.1–0.2em), tight on values. Labels whisper, values shout.
  • Numerals: font-variant-numeric: tabular-nums everywhere BPM, percentages, or beat counts appear. No jitter.

Design principles

Dark. Flat. Fluorescent.

Near-black surfaces, hair-thin borders, one glow color. The violet is the only source of light. Everything else gets out of its way.

One FORGE, one focus.

The device has exactly one primary action. The button is visually dominant before action, dims while running, returns on complete.

Data over chrome.

No skeuomorphic racks, no fake LCDs. The waveform, the spectroscope, the LEDs on the pads — those are the only ornaments, because they're telling the truth about audio.

B · Waveform + bar-grid display

The anchoring UI element. Shows the source clip's waveform with bar boundaries drawn from Ableton's warp markers, not from a rigid grid. This is the user's visual confirmation that warp analysis is correct before they forge.

  • Source: waveform~ fed by the clip file path via sfinfo~ / buffer~
  • Grid overlay: vertical 1px lines at every bar boundary computed from warp_markers + signature_numerator. Major bar (every 4 bars) is slightly brighter yellow.
  • Playhead: green 2px line driven by live.thisdevice → plugsync~ when parent clip is playing.
  • Color: waveform in violet #c084fc at 0.85 alpha; grid lines yellow #fbbf24 at 0.4 / 0.55; playhead green.
  • Top-right BPM readout: detected tempo + time signature. Click to toggle between clip BPM and session BPM readout.
  • Beat numbers underneath: bar-accurate, drawn from the warp-marker interpolation.
Fallback state If the clip isn't warped or get_warp_markers returns empty, the grid is drawn rigid at the session tempo and a small yellow triangle appears next to the BPM readout with tooltip: "Rigid grid — warp the clip for accurate bar detection". Do not refuse to forge; the librosa fallback still works.

C · Spectroscope

  • Source: spectroscope~. By default follows whichever stem is soloed in the mixer; otherwise the source clip.
  • Label: top-left, shows which signal is being analyzed.
  • Style: bars from deep-violet at bottom to light-violet at peak, matching the waveform gradient. 64 bins, log-spaced.
  • Secondary function: when a pad is held, the scope switches to showing that bar's average spectrum (still frame) so the user can see why the curator selected it.

E · Stem mixer (4 dials)

  • Objects: live.dial × 4, routed to per-stem *~ output gain on the device's internal stem-sum output.
  • Color coding: each dial's indicator line uses that stem's channel color. When the dial is at zero the color drops to 20% opacity.
  • Purpose is pre-split auditioning: after FORGE the device routes all four stems through its output so you can monitor a pre-mix before you commit the clips to the new tracks. MIDI-mappable.
  • Alt-click to solo: standard Ableton convention. Soloing a stem also switches the spectroscope feed.
Why it's here even though there are separate tracks The per-track gain in the Live session is downstream of the pipeline's heavy processing (bitcrush, reverb, etc.). The device-internal mixer is a dry-stem preview, useful to hear what the separation actually pulled out before deciding whether to re-forge with a different backend.

F · FORGE button

  • Object: textbutton with custom bgoncolor #7c3aed, bgcolor #1a1a1a (disabled), textcolor white.
  • States:
    • Disabled: background #1a1a1a, text #555, no glow. Shown when no source clip present.
    • Ready: full violet, glow 0 0 20px rgba(192,132,252,0.35).
    • Running: label swaps to CANCEL, background desaturates to #4a3878, kills Python subprocess on click.
    • Complete: label swaps back to FORGE with a subtle green border-flash (150ms) to confirm.
  • Keybind: none (devices can't register global keys), but MIDI-mappable.

G · Pipeline / Backend / Curation groups

  • Pipeline chips: 4 chips (default / idm_crushed / glitch / ambient) as selectable tabs. Active chip gets violet border + violet tinted background. Value mirrored in the group header label for redundancy.
  • Backend chips: 4 chips (auto / demucs / lalal / musicai). auto resolves to ft-fused locally. When lalal or musicai is selected, a tiny "$" balance indicator appears to the right of the group title (reads from stemforge balance).
  • Curation group: strategy (dropdown: max-diversity / rhythm-taxonomy / sectional), n_bars (live.numbox, 4–32, default 14), slice beats (toggle).

H · Curated pad grid

The v3 centerpiece and the reason the device is 780px wide, not 400.

  • Object: matrixctrl 4×4 + live.button overlay per cell for discrete LED control.
  • Per-pad data: each pad backs one entry in polybuffer~ sf_bars, fed by the curated *_beat_NNN.wav files. Extra pads beyond n_bars render as dimmed dots.
  • LED color = spectral centroid of that bar → warm (red) low, cool (teal/violet) high. Derived by the curator, stamped in manifest.
  • LED brightness = crest factor → punchy bars are brighter.
  • Click: one-shot trigger, plays through groove~ time-stretched to session tempo. Click + hold: loop until release. Shift-click: latch loop.
  • MIDI mapping: each pad is a live.button and is independently mappable. Push / Launchpad / keyboard all work. A "map all" button in the group header auto-maps to the 16 notes starting at C2.
  • Right-click on pad: opens contextual — "Drag to Simpler" (writes a drag-ref to the .wav), "Reject bar" (re-runs curator for a replacement), "Lock" (bar stays across re-curation).
IDM-specific behavior Held pads crossfade into each other via groove~'s release ramp (20ms), so mashing keys for glitchy retriggers doesn't click. The curator's rhythm-taxonomy strategy can be used to deliberately pick pads that are structurally-adjacent (same rhythm class), giving organized chaos.

I · Sync strip (footer)

  • SYNC LED: green pulsing while playback is in sync, red if groove~ reports drift > 5ms, yellow if the session tempo has drifted from the BPM inferred at forge time.
  • Tempo readout: BAR_BPM → SESSION_BPM. If they match, a single number. If different, both with arrow — pads are being stretched.
  • Strategy confirmation: right-side reminder of which curation strategy produced the current pad set (so you can A/B by re-running with a different one without forgetting which set you're listening to).

05Workflow — press to playback

The canonical happy path, end-to-end. Numbers match state transitions in §08 and the NDJSON events in v0/interfaces/ndjson.schema.json.

  1. User drops StemForge.amxd on any audio track. The track already has a source audio clip in slot 0 (warped by Ableton's auto-warp).
  2. Device boots. stemforge_lom.js walks live_set this_device canonical_parent clip_slots and finds the first populated slot. It reads file_path, signature_numerator, warp_markers, session tempo. The waveform and bar-grid render from this.
  3. User picks a pipeline (or accepts default) and toggles slice beats. n_bars and strategy live in the curation group.
  4. User presses FORGE. stemforge_bridge.js writes the LOM analysis to /tmp/sf-analysis-XXXX.json and spawns stemforge-native forge --analysis <json> --pipeline <p> --json-events. NDJSON flows back to stdout.
  5. Python emits started. Status strip flips to yellow, phase label reads LOADING_MODEL if cold (warmup should have prevented this on installed packages), else SPLITTING.
  6. ONNX + CoreML runs htdemucs_ft_fused. progress events fire roughly every 250ms, pushing percentage into the status bar. Spectroscope shows live input spectrum.
  7. Slicer runs. Using warp_markers + signature_numerator, per-beat sample positions are interpolated, grouped every numerator beats into bars. Slices are written to ~/stemforge/processed/{track}/{stem}_beats/beat_001.wav …
  8. Curator runs. Greedy max-diversity selects n_bars from all bars, writes curated/{stem}/, emits per-bar fingerprint metadata.
  9. Manifest emitted. stems.json holds: bpm, track_name, pipeline, per-stem wav_path + beats_dir + curation, plus processed_at timestamp.
  10. complete event. Bridge reads the manifest path, hands it to stemforge_loader.js.
  11. Loader sets session tempo from manifest.bpm via LiveAPI("live_set").set("tempo", bpm). This is the moment the session adopts the source's clock.
  12. Loader places clips. Per the rules in §06 — find-or-duplicate template tracks per stem_target, call create_audio_clip, set warp_mode from the pipeline config, looping=1.
  13. Beat-chop path. The drums _beats/ directory is passed to the duplicated SF | Beat Chop Simpler track's Simpler (slice mode, warp off).
  14. Curated bars hydrate polybuffer~ in the device. When done, pad LEDs light up and the grid becomes playable.
  15. Device flips to COMPLETE. Green dot, grid ready, stem mixer live, waveform still showing source with playhead active when source is playing.

Templates (factories)

The placement rules, explicit

  1. Template lookup — for each stem in the manifest, match against tracks.yaml stem_target. If the pipeline's template field overrides (e.g. idm_crushed routes drums → "SF | Drums Crushed" instead of Drums Raw), that wins.
  2. Track duplication — LiveAPI("live_set").call("duplicate_track", templateIndex). The new track lands at templateIndex + 1. Known limitation: Live does not let us insert tracks at arbitrary indices from JS, so post-forge track order is template, copy, template, copy, … not grouped by type. A follow-up enhancement is post-hoc grouping via group_track — traceable in a future W-work item.
  3. Rename + recolor — new track becomes {source_track_name} · {stem}, color set from pipeline YAML.
  4. Effect hydration — applyEffects() walks pipeline.stems[stem].effects and sets device parameters by name on the pre-existing chain devices (the templates already have their VSTs).
  5. Clip placement — clip_slots[0].call("create_audio_clip", wav_path). Slot 0 by default; if occupied, walk slots for first empty.
  6. Warp mode per stem — clip.set("warp_mode", WARP_MODES[pipeline.warp_mode]) — beats=0 for drums/bass, tones=1 for vocals, complex=4 or complex-pro=5 for textures.
  7. Loop — clip.set("looping", 1). Every placed clip loops by default — this is an IDM-curation tool, you always want looping.
  8. Beat-chop path — SF | Beat Chop Simpler (MIDI) is duplicated, named {track} | chop, the duplicated Simpler's sample source is replaced with {drums_beats_dir} in slice mode, warp off. Slice count = bar-count (one slice per bar).

Multi-forge choreography

When a user forges a second track (different source), the device detects the existing stem-routed tracks and does not re-duplicate the templates. Instead, it appends a new clip slot to each already-duplicated track:

  • First forge: clips land in slot 0
  • Second forge: clips land in slot 1 (same tracks, new scene)
  • Third forge: slot 2, and so on

This means scenes line up horizontally across all stem tracks — press a scene launch and you're auditioning one forge's complete stem set. Session tempo is not re-set on subsequent forges (first forge wins); curated bars from forge 2 will time-stretch to forge 1's tempo via groove~.

The "shared tempo across forges" story This is exactly why the device-internal pad grid uses groove~ rather than playing at native speed: so you can pull breaks from a 94 BPM funk record and an 170 BPM d&b record and play them interleaved at the session's 128 BPM, all on the same grid. The session BPM is the canonical clock; everything else bends.

The three anchor points

  1. Source clip BPM — computed by Ableton's warp engine when the user drops audio. Read via LiveAPI("live_set").get("tempo") at forge-time. This is the authority for the source's intended tempo.
  2. Session BPM — set by the loader via setBPM(manifest.bpm). On first forge, session tempo adopts the source's BPM. On subsequent forges against the same session, session tempo is not overwritten — the user is curating across tempos and wants stability.
  3. Pad-bar BPM — each curated bar carries its source tempo as manifest metadata. When a pad triggers, groove~ computes the stretch ratio session / bar and plays the bar at session tempo. No clicks, no phase weirdness, because groove~'s interp 1 mode handles small ratios cleanly.

The grid contract

All curated bars are exactly signature_numerator beats long as measured at the bar's source tempo. Because warp analysis gave us bar boundaries in sample positions, the .wav is trimmed sample-accurate to the bar. When groove~ stretches it to session tempo, the result lines up to the session's 1 / 2 / 3 / 4 grid without drift, regardless of source tempo or source time-signature.

This is why 7/4 tracks work: the bar is simply seven beats long at source tempo. The pad plays it as a 7-beat loop at session tempo; if session is in 4/4, the loop repeats every 1.75 measures — audibly polyrhythmic but rhythmically honest. Users hear the groove the way JB intended, projected into their host rhythm.

08State machine & feedback

10v0 → v3 migration

You are shipping v0 this week. This spec describes v3. The plan for getting from here to there without re-work:

v0 subset (ships now, 400 × 260)

  • Titlebar, file-drop target, pipeline + backend dropdowns, slice toggle, progress bar, status text, Split button — exactly as device.yaml specifies.
  • Clip placement via stemforge_loader.v0.js — working today per v0-ableton-validation.md.
  • Set tempo from manifest — working today.
  • All of §06 (where clips land) and §07 (tempo contract) work against v0 already. They're session-level behavior; the device UI surface is what grows.

v1 — device grows to 780 × 340

  • Add waveform display with warp-aware bar grid. This unlocks the "confirm warp before forge" moment.
  • Replace progress-bar + status line with the richer status strip.
  • Refactor dropdowns as chip selectors (same underlying live.menu, re-skinned).
  • Still no pad grid, still no mixer — clips in the session are your pads.

v2 — stem mixer + spectroscope
